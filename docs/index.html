DeskNest — a Germany-first WooCommerce store, built and documented like a real project.

DeskNest is a focused desk-setup and workspace-accessories store concept for remote professionals, developers, freelancers, and students. It was implemented locally with WordPress, WooCommerce, and a custom classic PHP theme — then reviewed, debugged, and documented scope by scope.

Where you are: GitHub Pages hosts only this static presentation. WordPress and WooCommerce do not run here, and there is no public live store demo or production deployment. The screenshots and facts below come from the local implementation and its repository documentation.

Commercial concept

DeskNest sells practical, well-designed desk-setup and productivity accessories: ergonomic essentials, desk organization, lighting and ambience, cable management, productivity accessories, and curated bundles — six commercial categories priced in EUR for a Germany-first market.

The concept is intentionally narrow. Product mix, pricing, category structure, copy, and shipping assumptions were designed around one credible commercial niche rather than a generic tutorial store.

Target customers

  • Remote and hybrid professionals
  • Developers and multi-device workspace users
  • Freelancers and independent creators
  • Students and home-office users
  • Germany-first buyers, with EU-friendly future positioning

Implementation highlights

Everything below was implemented and validated in the local WooCommerce environment and is recorded in the repository's scope documentation.

Catalog & Inventory

  • 6 commercial categories, 24 published products with EUR pricing and a consistent SKU strategy.
  • Quantity tracking on all 19 simple products and 11 variations.
  • Intentional low-stock and out-of-stock scenarios for realistic storefront states.

Attributes & Variations

  • 3 global attributes with 9 terms.
  • 5 variable products with 11 variations, each with its own SKU, price, and stock.
  • Selected-variation workflow with updated SKU, price, stock, and add-to-cart state.

Storefront UX

  • Custom classic PHP theme desknest-storefront — no page builder.
  • theme.json design tokens, reusable template parts, and selected WooCommerce template overrides.
  • Responsive navigation, product cards, and shop/category archives.

Cart & Checkout

  • Classic Cart implemented through a styled WooCommerce template override.
  • Checkout Block integrated and styled with scoped, block-compatible CSS.
  • 4 realistic local coupons configured and validated.

Customer & Operations

  • Registration, login, dashboard, orders, addresses, and account-details states styled locally.
  • Order-management workflow validated against one development order.
  • 7 local-safe product reviews; local-development-only BACS payment method; one Germany-only shipping zone with flat rate and a free-shipping threshold.

QA & Documentation

  • Structured QA pass: static checks, route checks, responsive and keyboard checks, and regression retests.
  • 4 confirmed storefront defects fixed and retested locally.
  • Scope-by-scope Markdown documentation covering architecture, configuration, and evidence.

Technical architecture

The stack below describes the local application. The page you are reading is a separate, framework-free static presentation — plain HTML, CSS, and a small amount of vanilla JavaScript.

  • WordPress 7.0
  • WooCommerce 10.9.3
  • PHP 8.2.29
  • MySQL 8.4.0
  • HTML5
  • CSS3
  • Vanilla JavaScript (minimal)
  • Git & GitHub
  • LocalWP on Windows 11

The custom theme lives at app/public/wp-content/themes/desknest-storefront in the repository. WordPress core and the WooCommerce plugin are intentionally not tracked.

Honest project boundaries

DeskNest is a local WordPress/WooCommerce portfolio implementation. GitHub Pages hosts only this static presentation — WordPress and WooCommerce do not run on GitHub Pages. To keep the record honest:

  • No public live store demo and no production deployment exist.
  • No real payment processing and no real bank credentials — the single BACS method is local-development-only.
  • No real customer data; reviews and the one order are local development data.
  • No carrier integration or shipment fulfilment — shipping is a local configuration.
  • No CI/CD, GitHub Actions, or Docker workflow.
  • No WCAG certification or full accessibility compliance claim — accessibility work was focused keyboard/focus review.
  • No penetration testing or production security certification — security work was a local hardening review.
  • No production Lighthouse, PageSpeed, or Core Web Vitals scores — performance work was a LocalWP baseline and asset review.
